# pip-tools & pip-compile — Complete Reference Guide

## What is pip-tools?

`pip-tools` is a package that gives you **two commands**:

| Command | Purpose | Analogy |
|---------|---------|---------|
| `pip-compile` | Reads your `.in` file → generates a locked `.txt` file | Like `npm install` generating `package-lock.json` |
| `pip-sync` | Installs exactly what's in the `.txt` file (removes extras too) | Like `npm ci` — clean, reproducible install |

### The Core Problem It Solves

Without pip-tools, you have two bad options:

1. **`pip freeze > requirements.txt`** — dumps everything (including sub-dependencies you never asked for). You lose track of what YOU actually need vs what was pulled in automatically.
2. **Hand-written `requirements.txt`** — you forget sub-dependencies, and different installs may resolve to different versions. Builds are not reproducible.

**pip-tools gives you both**: you write what you *want*, it generates what you *need*.

### Framework Comparison

```
Spring Boot:  build.gradle (you write)  →  gradle.lockfile (generated)
Laravel:      composer.json (you write)  →  composer.lock (generated)
Node/Express: package.json (you write)   →  package-lock.json (generated)
Python:       requirements.in (you write) →  requirements.txt (generated)
                    ↑ pip-compile does this transformation
```

## Installation

```bash
# Install pip-tools globally (or in your virtualenv)
pip install pip-tools

# Verify
pip-compile --version
pip-sync --version
```

> **Best Practice**: Install pip-tools inside your virtual environment so it uses the correct Python version for dependency resolution.

```bash
# Full setup from scratch
python3 -m venv venv
source venv/bin/activate          # macOS/Linux
pip install pip-tools
```

## Project Structure — The Full Setup

Yes, you **absolutely should** have `.in` files for development, production, and test too — not just base. Here's the complete corporate-grade structure:

```
requirements/
├── base.in              # What YOU depend on (core packages)
├── base.txt             # Generated by pip-compile (locked, full tree)
├── development.in       # Dev-only dependencies
├── development.txt      # Generated
├── production.in        # Production-only dependencies
├── production.txt       # Generated
├── test.in              # Test-only dependencies
└── test.txt             # Generated
```

### Why Have `.in` Files for Every Environment?

This was your exact question — great one. The reason is **the same reason you split settings into base/dev/prod**:

1. **Clarity** — You can instantly see "what did I *actually* choose to install for dev?" without wading through 200 transitive dependencies
2. **Auditability** — In banking apps, security teams review dependencies. Showing them a 5-line `.in` file is far better than a 150-line `.txt` file
3. **Upgrade control** — When you want to upgrade `Django`, you change ONE line in `base.in` and recompile. pip-compile figures out all the cascading sub-dependency changes
4. **Separation of concerns** — `debug-toolbar` should never appear in production, `gunicorn` should never appear in dev. The `.in` files make this intent crystal clear

## Step-by-Step: Setting Up pip-tools

### Step 1: Create the `.in` files

**`requirements/base.in`** — Your actual, direct dependencies:
```
# Core
Django>=5.1,<6.0
djangorestframework>=3.15,<4.0

# Database
psycopg2-binary>=2.9

# Configuration
django-environ>=0.11

# Models
django-model-utils>=4.5

# CORS
django-cors-headers>=4.4
```

**`requirements/development.in`** — Inherits base, adds dev tools:
```
-r base.in

# Debugging
django-debug-toolbar>=4.4
ipython>=8.27

# Code Quality
black>=24.8
flake8>=7.1
isort>=5.13
pre-commit>=3.8

# Hot Reload
watchfiles>=0.24
```

**`requirements/test.in`** — Inherits base, adds test libraries:
```
-r base.in

# Testing Framework
pytest>=8.3
pytest-django>=4.9
pytest-cov>=5.0

# Fixtures & Fake Data
factory-boy>=3.3
faker>=30.0

# API Testing
pytest-drf>=1.1
```

**`requirements/production.in`** — Inherits base, adds production needs:
```
-r base.in

# WSGI Server
gunicorn>=23.0

# Caching
django-redis>=5.4
redis>=5.0

# Monitoring
sentry-sdk[django]>=2.14

# Security
django-csp>=3.8

# Static Files
whitenoise>=6.7
```

### Step 2: Compile (Generate the `.txt` lock files)

```bash
# Compile each .in → .txt
pip-compile requirements/base.in        --output-file requirements/base.txt
pip-compile requirements/development.in --output-file requirements/development.txt
pip-compile requirements/test.in        --output-file requirements/test.txt
pip-compile requirements/production.in  --output-file requirements/production.txt
```

> **Order matters**: Compile `base.in` first, since others depend on it via `-r base.in`.

### Step 3: See What pip-compile Generates

After running `pip-compile requirements/base.in`, the generated `base.txt` looks like:

```
#
# This file is autogenerated by pip-compile with Python 3.12
# by the following command:
#
#    pip-compile requirements/base.in
#
asgiref==3.8.1
    # via django
django==5.1.2
    # via
    #   -r requirements/base.in
    #   django-cors-headers
    #   django-model-utils
    #   djangorestframework
django-cors-headers==4.4.0
    # via -r requirements/base.in
django-environ==0.11.2
    # via -r requirements/base.in
django-model-utils==4.5.1
    # via -r requirements/base.in
djangorestframework==3.15.2
    # via -r requirements/base.in
psycopg2-binary==2.9.9
    # via -r requirements/base.in
sqlparse==0.5.1
    # via django
```

Notice how every package has a **comment showing WHY it's there**:
- `asgiref` → because `django` needs it
- `sqlparse` → because `django` needs it
- `django` → because YOU asked for it in `base.in` AND other packages need it

This is incredibly valuable for auditing and debugging dependency conflicts.

### Step 4: Install Dependencies

```bash
# For local development
pip-sync requirements/development.txt

# For running tests (CI)
pip-sync requirements/test.txt

# For production deployment
pip-sync requirements/production.txt
```

> **`pip-sync` vs `pip install -r`**: `pip-sync` is stricter — it removes packages NOT in your requirements file. This ensures your environment matches exactly what's declared. `pip install -r` only adds, never removes.

## Daily Workflow — Common Operations

### Adding a New Package

```bash
# 1. Add to the appropriate .in file
echo "celery>=5.4" >> requirements/base.in

# 2. Recompile
pip-compile requirements/base.in --output-file requirements/base.txt

# 3. Since base changed, recompile dependent files too
pip-compile requirements/development.in --output-file requirements/development.txt
pip-compile requirements/test.in        --output-file requirements/test.txt
pip-compile requirements/production.in  --output-file requirements/production.txt

# 4. Sync your environment
pip-sync requirements/development.txt
```

### Upgrading a Specific Package

```bash
# Upgrade only Django to latest allowed by your constraint
pip-compile --upgrade-package django requirements/base.in --output-file requirements/base.txt

# Upgrade all packages to latest allowed versions
pip-compile --upgrade requirements/base.in --output-file requirements/base.txt
```

### Removing a Package

```bash
# 1. Remove the line from the .in file
# 2. Recompile — pip-compile will drop it and its orphan sub-dependencies
pip-compile requirements/base.in --output-file requirements/base.txt

# 3. Sync — pip-sync will uninstall the removed packages
pip-sync requirements/development.txt
```

### Checking for Outdated Packages

```bash
# See what can be upgraded
pip list --outdated
```

## Makefile — Automate Everything (Corporate Best Practice)

Create a `Makefile` at your project root to avoid remembering all these commands:

```makefile
.PHONY: compile-deps sync-deps upgrade-deps

# Compile all .in files to .txt
compile-deps:
	pip-compile requirements/base.in        --output-file requirements/base.txt
	pip-compile requirements/development.in --output-file requirements/development.txt
	pip-compile requirements/test.in        --output-file requirements/test.txt
	pip-compile requirements/production.in  --output-file requirements/production.txt

# Sync your local dev environment
sync-deps:
	pip-sync requirements/development.txt

# Upgrade all packages to latest allowed versions
upgrade-deps:
	pip-compile --upgrade requirements/base.in        --output-file requirements/base.txt
	pip-compile --upgrade requirements/development.in --output-file requirements/development.txt
	pip-compile --upgrade requirements/test.in        --output-file requirements/test.txt
	pip-compile --upgrade requirements/production.in  --output-file requirements/production.txt
	pip-sync requirements/development.txt
```

Now your team just runs:

```bash
make compile-deps    # After editing any .in file
make sync-deps       # After pulling changes (like npm install)
make upgrade-deps    # Monthly dependency upgrade cycle
```

## Git Strategy — What to Commit

```
✅ COMMIT these (always):
   requirements/base.in
   requirements/development.in
   requirements/test.in
   requirements/production.in
   requirements/base.txt
   requirements/development.txt
   requirements/test.txt
   requirements/production.txt

❌ NEVER COMMIT:
   venv/
   .venv/
```

**Why commit the `.txt` files?** Because they're your lock files. Every developer and CI/CD pipeline should use the exact same resolved versions. This is identical to committing `package-lock.json` in Node or `composer.lock` in Laravel.

## CI/CD Integration (Docker Example)

```dockerfile
# In your Dockerfile for production
FROM python:3.12-slim

WORKDIR /app

# Install dependencies first (Docker layer caching)
COPY requirements/production.txt requirements/production.txt
RUN pip install --no-cache-dir -r requirements/production.txt

# Then copy app code
COPY . .
```

```yaml
# In your GitHub Actions CI for testing
- name: Install test dependencies
  run: |
    pip install pip-tools
    pip-sync requirements/test.txt
```

## Quick Command Cheatsheet

| Task | Command |
|------|---------|
| Install pip-tools | `pip install pip-tools` |
| Compile .in → .txt | `pip-compile requirements/base.in --output-file requirements/base.txt` |
| Sync environment | `pip-sync requirements/development.txt` |
| Upgrade one package | `pip-compile --upgrade-package django requirements/base.in -o requirements/base.txt` |

| Check outdated | `pip list --outdated` |
| Generate hashes (security) | `pip-compile --generate-hashes requirements/base.in -o requirements/base.txt` |
| Dry run (preview changes) | `pip-compile --dry-run requirements/base.in` |

## pip-tools vs Alternatives

| Feature | pip-tools | Poetry | Pipenv |
|---------|-----------|--------|--------|
| Lock file | `.txt` (pip-native) | `poetry.lock` | `Pipfile.lock` |
| Learning curve | Low (just pip) | Medium | Medium |
| Speed | Fast | Medium | Slow |
| Corporate adoption | Very high | Growing | Declining |
| Works with `pip install -r` | Yes (`.txt` is standard) | No (needs poetry) | No (needs pipenv) |
| Django community preference | ✅ Most common | ✅ Also popular | ❌ Less common |

**Why pip-tools wins for corporate/banking projects**: The generated `.txt` files are standard pip format. Any server, any Docker image, any CI system can use them with plain `pip install -r`. No extra tooling needed in production. Poetry and Pipenv require their own runtimes to be installed everywhere.

## Key Takeaway

```
.in files  = YOUR intent     (what you chose to depend on)
.txt files = RESOLVED truth  (exact versions of everything, including sub-deps)

You edit .in files.
pip-compile generates .txt files.
pip-sync installs from .txt files.

That's it. Three concepts, bulletproof dependency management.
```
